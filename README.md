# AIC Token Estimator

> A deliverable from Lantern for FinOps and Developer Platform teams.

A Python CLI — the **AIC Token Estimator** — that converts GitHub Copilot **AI Credits** from a monthly AI Usage Report CSV into estimated token counts (input / cached / cache-write / output / total) per user, per model, per time period.

## Why this exists

GitHub prices every Copilot interaction in **tokens**, sums the cost in dollars, divides by `$0.01`, and exposes only the final **AI Credit** number in the AI Usage Report and Billing API. Tokens are destroyed in that compression — one credit is not directly invertible to a unique token count because it collapses input, cache-read, cache-write, and output into one scalar.

**The unlock:** the AI Usage Report splits credits by **model and by user**. Because per-model token rates differ by 10–150×, decomposing credits by model first removes the dominant source of error. From there the tool inverts each model's credits using calibrated mix assumptions and reports a confidence band.

This gives FinOps a normalized **token + USD basis** to measure Copilot alongside other AI services (Claude API, Codex, etc.).

## Capabilities

| Capability | Description |
|---|---|
| **XLSX input** | Accepts `.xlsx` files directly in addition to `.csv` — no manual export needed (standard, unencrypted xlsx only) |
| **Credit → token inversion** | Converts AI Credits back to estimated input / cached / cache-write / output / total tokens per user per model |
| **Confidence bands** | Reports analytically derived `low` and `high` token bounds, not just a point estimate |
| **Auto-select detection** | Handles `Auto: <Model>` prefix in the CSV — applies GitHub's 10% auto-select discount automatically |
| **Multi-period bucketing** | Buckets results by `day`, `week`, or `month` for trend analysis |
| **Model pivot tables** | Produces chart-ready CSVs (model × period) ready to drop into Excel or Power BI |
| **Executive summary** | Generates a Markdown leadership snapshot with top models, top users, and period trend |
| **Billing API fetch** | Optionally pulls the usage report via GitHub's Enhanced Billing API instead of a manual download |
| **Built-in selftest** | Six deterministic checks verify the math and setup with no external data required |
| **No dependencies** | Python standard library only — no `pip install` of third-party packages |
| **Offline capable** | Runs entirely locally; no internet connection required after setup |

## Quick start

```bash
# Place your AI Usage Report CSV in copilot_usage_reports/<YYYY-month>/
python tools/aic_token_estimator/aic_tokens.py estimate \
  --csv copilot_usage_reports/2026-april/ai-usage-report.csv \
  --period month \
  --reports-dir token_estimate_reports \
  --report-stem token-estimate
```

Reports are written to `token_estimate_reports/2026-04/` (year-month derived from the data).

## CLI reference

### `estimate` — convert credits to tokens

```
python tools/aic_token_estimator/aic_tokens.py estimate [options]
```

| Flag | Required | Description |
|---|---|---|
| `--csv <path>` | Yes | Path to the AI Usage Report CSV or XLSX |
| `--period day\|week\|month` | No (default: `month`) | Time bucket granularity |
| `--reports-dir <path>` | No (default: `token_estimate_reports`) | Base folder for output artifacts |
| `--report-stem <name>` | No (default: derived from CSV filename) | Base filename for report artifacts |
| `--out <path>` | No | Write the detailed CSV to a specific path |
| `--summary-out <path>` | No | Write the executive summary to a specific path |
| `--cache-fraction <C>` | No | Pin the cache-read fraction of input tokens (0–1) for all rows to reconcile the estimate against a measured token-throughput report. Higher `C` → more tokens per credit |
| `--output-ratio <RHO>` | No | Pin the output:input token ratio for all rows (calibration knob; pairs with `--cache-fraction`) |
| `--cache-write-fraction <W>` | No | Pin the cache-write fraction (0–1); needed only for ultra-cache-heavy tenants where `--cache-fraction` exceeds the prior cache-write floor |

**`--period` values:**

| Value | Bucket key | Output subdir | Use for |
|---|---|---|---|
| `day` | `YYYY-MM-DD` | `YYYY-MM-DD` | Single-day or daily token usage. When the data covers exactly one day and no `--report-stem` is given, the date is appended to all output filenames automatically. |
| `week` | `YYYY-Www` (ISO, Monday-start) | `YYYY-MM` | Weekly token usage by model |
| `month` *(default)* | `YYYY-MM` | `YYYY-MM` | Monthly leadership roll-up |

Eliminates the manual CSV download step. Requires an enterprise admin or billing-manager token.

Then pass the output to `estimate` as normal. The API returns the same credit/dollar data as the manual CSV — tokens are still estimated locally.

## Output artifacts

Each `estimate` run writes five files to `<reports-dir>/<YYYY-MM>/`:

| File | Contents |
|---|---|
| `<stem>.csv` | Detailed per `user × period × model` rows with credits, USD cost, token split, and confidence band |
| `<stem>-model-by-<period>.csv` | **Chart-ready pivot**: rows = period bucket, columns = model, cells = estimated tokens |
| `<stem>-model-by-<period>-usd.csv` | Same pivot in exact USD (reconciles to the penny with billing portal) |
| `<stem>-tokens-by-user.csv` | Per-user totals across all models (credits, USD, total tokens, low/high bounds), sorted by cost |
| `<stem>-executive-summary.md` | Leadership snapshot: top models, top users, period trend, model-over-time matrix |

### Detailed CSV columns

| Column | Description |
|---|---|
| `username` | GitHub username |
| `month` / `date` / `week` | Period bucket (header varies with `--period`) |
| `model` | Canonical model ID (e.g. `claude-sonnet-4.6`) |
| `credits` | AI credits charged (gross, from GitHub report) |
| `cost_usd` | Exact cost — `credits × $0.01` |
| `est_input_tokens` | Estimated uncached input tokens |
| `est_cached_tokens` | Estimated cache-read tokens |
| `est_cache_write_tokens` | Estimated cache-write tokens (Anthropic only) |
| `est_output_tokens` | Estimated output tokens |
| `est_total_tokens` | Estimated total tokens (expected point) |
| `total_tokens_low` | Lower bound of confidence band |
| `total_tokens_high` | Upper bound of confidence band |
| `unresolved_credits` | Credits for unrecognized models |

### Reconciling to the billing portal

Sum `cost_usd` in the output CSV — it should match your portal's gross amount to the penny. Credits are exact; only the token *split* is estimated.

> **Use gross credits, not net.** Net credits reflect billing after the included seat pool. Within the pool, `net_amount` is often `$0` even though millions of tokens were consumed. Always use `aic_quantity` / `gross_amount`.

## Design overview

```
copilot_usage_reports/<period>/      tools/aic_token_estimator/
        ai-usage.csv            →         aic_tokens.py (CLI)
                                               │
                              ┌────────────────┼────────────────┐
                              ▼                ▼                ▼
                         catalog.py       estimator.py    billing_api.py
                     (pricing table)    (credit→token      (fetch from
                                          inversion)       GitHub API)
                              │
                              ▼
              token_estimate_reports/<YYYY-MM>/
                  <stem>.csv
                  <stem>-model-by-<period>.csv
                  <stem>-model-by-<period>-usd.csv
                  <stem>-executive-summary.md
```

**Two-phase pipeline:**

1. **Aggregate** — stream the input CSV, sum credits by `(user, bucket, model, sku)`. Reduces N rows to O(unique aggregates) before any math.
2. **Invert** — call `estimate_tokens()` once per aggregate using each model's published rates and SKU-specific mix priors.

**Key design decisions:**

- `catalog.py` is the single source of truth for all pricing. Updating GitHub's rates only requires changing this file.
- `estimator.py` applies `AUTO_DISCOUNT = 0.90` when the model name has an `Auto:` prefix — GitHub charges 10% fewer dollars under auto-select, so the same credits represent ~11.1% more tokens.
- Confidence bands are computed **analytically** (not by sampling) by sweeping the output:input ratio `ρ` and cache fraction `c` across SKU-specific plausible ranges.
- All dependencies are Python standard library — the tool runs fully offline with no network access after `pip install -e .`.

**SKU mix priors** (`estimator.py`):

| SKU | ρ (out:in) low–exp–high | Cache frac low–exp–high |
|---|---|---|
| `coding_agent_ai_credit` (agent/edit) | 0.04 – 0.12 – 0.30 | 0.55 – 0.85 – 0.97 |
| `copilot_ai_credit` (chat, review) | 0.10 – 0.25 – 0.50 | 0.30 – 0.55 – 0.80 |
| `default` (unknown SKU) | 0.05 – 0.20 – 0.45 | 0.40 – 0.70 – 0.90 |

## Contributing & guidelines

This repo uses instruction files in `.github/instructions/` that Copilot and contributors must follow. Read the relevant file before modifying code of that type:

| File type | Instruction file |
|---|---|
| `*.py` | `.github/instructions/python.instructions.md` |
| `*.md` | `.github/instructions/markdown.instructions.md` |
| `*.csv` | `.github/instructions/csv.instructions.md` |

**Key rules:**

- **Always use the full package path** from repo root: `from tools.aic_token_estimator.catalog import ...`
- **Never hardcode `0.01`** — always import and use `USD_PER_CREDIT` from `catalog.py`.
- **No linter, formatter, or test framework** is configured. The only automated check is `selftest`.
- **Generated files** (`token_estimate_reports/<YYYY-MM>/`) must not be hand-edited. Re-run `estimate` to regenerate.
- The `pyproject.toml` version follows the package version in `aic_token_estimator.egg-info/`. Bump both together.

## FAQ

**Q: Why is `cost_usd` exact but token counts are estimates?**
GitHub reports credits (which map exactly to dollars at `$0.01` each), but never exposes the token breakdown. The tool inverts the dollar amount back to tokens using published per-model rates and calibrated mix assumptions.

**Q: Why are these token numbers lower than GitHub's "token usage by user" report?**
They measure two different things, and both are valid. This tool inverts **dollars** back into tokens, so it reports **cost-weighted (billable) tokens**: GitHub bills cache-read tokens at ~10% of the uncached-input rate, so cache reads barely move the dollar total and therefore barely register in a dollar-anchored inversion. GitHub's metrics report counts **raw throughput** — every token at 100%, including cache reads (which agentic sessions re-read on every iteration) and possibly IDE code completions (which carry *no* AI credits at all and so can never be recovered from credits). For cache-heavy agentic usage the throughput view can run several times higher. To make this tool reproduce a throughput figure, re-run with `--cache-fraction <c>` (and, for very cache-heavy tenants, `--output-ratio` and `--cache-write-fraction`) pinned to your measured values — the dollars stay identical to the penny; only the token split moves.

**Q: Do code completions appear in the report?**
No. IDE code completions are billed separately and do not appear as AI Credits. This tool only covers interactions that consume AI Credits (chat, agent, code review, etc.).

**Q: How often should I run this?**
Monthly. The AI Usage Report CSV covers at most 31 days — download and run separately for each billing period.

**Q: Why does `net_amount` often show `$0`?**
Within the included seat pool, usage is covered by the pool so the net charge is `$0` even though tokens were consumed. Always use gross credits (`aic_quantity`) for token estimation.

**Q: What does `Auto:` mean in the model column?**
When a user lets Copilot auto-select the model, the CSV shows `Auto: <Model Name>`. The tool strips the prefix, looks up the actual model's rates, and applies a 10% discount (GitHub charges 10% fewer dollars under auto-select, so the same credits correspond to ~11.1% more tokens).

**Q: What do I do if I see `UNRESOLVED credits` in the output?**
A model name in the CSV was not recognized by the pricing catalog — usually a newly released model. Record the exact model name from the CSV and send it to Lantern to update `catalog.py`.

**Q: Is the egg-info folder required?**
It is generated automatically by `pip install -e .` and is required for the `tools.*` import path to resolve. It is gitignored and should never be committed — just re-run `pip install -e .` after a fresh clone.

**Q: Can I run this without internet access?**
Yes. All code uses Python standard library only. After the one-time `pip install -e .`, no internet connection is needed.

## Troubleshooting

**`error: … appears to be an encrypted .xlsx or a legacy .xls file`**
The file has been password-protected in Excel, or it is an old `.xls` file saved with a `.xlsx` extension. Open it in Excel, remove any password protection (File → Info → Protect Workbook), and re-save as `.xlsx` — or export directly to CSV.

**`ModuleNotFoundError: No module named 'tools'`**
The virtual environment is not activated, or `pip install -e .` was not run. Repeat the [Installation](#installation) steps.

**`python: command not found`**
Use `python3` instead of `python`, or ensure Python 3.10+ is on your `PATH`.

**`error: CSV not found`**
Check that the `--csv` path is correct relative to the repo root and that the file exists in `copilot_usage_reports/<period>/`.

**`selftest` fails**
Do not proceed. Contact Lantern with the full error output.

**`UNRESOLVED credits` warning**
See FAQ above. The estimate will still run; unresolved credits are reported separately and excluded from token totals.
